## 1. 背景・目的

Phase 4-B として「イベント・観光スポット・グルメスポットを横断検索できるUI」の実装を計画していたが、3モデルのエリア区分が以下のようにバラバラであることが判明した。

| モデル | エリア区分 | 粒度 |
|---|---|---|
| Event | `area`（13市町村＋その他） | 市町村単位（一部のみ） |
| Spot | `area`（県北・県中・県南・会津・南会津・いわき・相双の7区分） | 地方単位 |
| Restaurant | `area`（Spotと同じ7区分） | 地方単位 |

横断検索でエリアをまたいだ一貫性のある絞り込みを行うには、この不一致を先に解消する必要がある。これが本設計のスコープ（サブプロジェクトA）。

横断検索UI自体（サブプロジェクトB）は本設計の対象外で、Aの実装完了後に別途設計する。

### 1-1. 調査で判明した事実

1. **正式な59市町村リストが既に存在する**
   `backend/app/services/municipality_detector_service.rb` の `MUNICIPALITIES` 定数に、福島県の全59市町村が定義済み。これは「制覇マップ」機能（`frontend/public/fukushima.geojson`、`FukushimaMap.tsx`、`VisitRecord.municipality`）が使っているのと同一の市町村セットであり、既に本アプリの事実上の正式エリアマスタとして機能している。

2. **Spot / Restaurant には既に `municipality` カラムが存在し、データも綺麗**
   スキーマ上 `municipality`（string, nullable, バリデーションなし）が既に存在し、調査の結果 Spot 30件・Restaurant 27件のすべてで、59市町村リストに含まれる有効な値が入力済み（空欄0件・無効値0件）。データ移行は不要で、バリデーション追加のみで済む。

3. **フロントエンドのイベント検索UIは既に59市町村ベースで作られていた**
   `frontend/src/components/events/FilterDrawer.tsx` の `AREA_GROUPS` は中通り・浜通り・会津の3グループで56市町村を選択肢として持つ（正式リストとの差分は桑折町・国見町・飯舘村の3件欠落のみ）。一方バックエンドの `Event::AREAS` は13市町村＋その他に制限されているため、**「会津若松市」等で検索しても該当イベントが存在し得ない潜在バグ**が既にあった。今回の統一はこのバグ修正を兼ねる。

4. **`AreaSelectMap.tsx`（スポット検索の地図UI）は既に59市町村ベースで実装済み**
   変更不要。

5. **Event の既存データで `area = 'その他'` は363件中11件**
   内容を確認した結果、1件は本文に「会津若松市」が含まれ自動判定可能、残り10件は「霊まつり流灯花火大会」「圓藏寺」「あいべこ」など柳津町（やないづまち）特有のキーワードを含む同一シリーズのイベントで、手動判定により柳津町と特定できた。

## 2. 設計方針

### 2-1. エリアマスタの一本化

`MunicipalityDetectorService::MUNICIPALITIES`（59市町村）を、Event / Spot / Restaurant 共通の正式なエリア値リストとする。新しい定数を作るのではなく、既存のこのリストを単一の参照元（single source of truth）として扱う。

- **Event**: `AREAS` を13市町村から59市町村に拡張する。加えてオンライン開催イベント用に `オンライン` を独立した特殊値として維持する（市町村ではないため59件には含めない）。`その他` は廃止する（全イベントが具体的な市町村に分類されるため不要になる）。
- **Spot / Restaurant**: `municipality` カラムに `presence: true, inclusion: { in: MunicipalityDetectorService::MUNICIPALITIES }` のバリデーションを追加する。

### 2-2. `area`（7地方区分）列の扱い

Spot / Restaurant の `area`（県北・県中・県南など7地方区分）列は**削除せず、現状のまま残す**。地方単位の表示（バッジ等）に使われており、横断検索のスコープには直接関係しないため、今回は触らない。

### 2-3. Event 側のデータ移行

`area = 'その他'` の11件を、調査済みの判定結果に従って手動で更新する。

| 件数 | 移行先 | 根拠 |
|---|---|---|
| 1件（id=416） | 会津若松市 | 本文に「会津若松市 鶴ヶ城集合」と明記 |
| 10件（id=467〜476） | 柳津町 | 圓藏寺・あいべこ・霊まつり流灯花火大会など柳津町特有の固有名詞を含む同一シリーズ |

データ修正用の通常のRailsマイグレーション（`up`/`down` を持つデータ移行マイグレーション）として実装する。rake taskにすると本番デプロイ時に手動実行を忘れるリスクがあるのに対し、マイグレーションなら既存のデプロイ手順（`bundle exec rails db:migrate RAILS_ENV=production`）に乗って自動的に反映される。タイトル文字列で対象行を特定し、`update_all` でバリデーションを経由せず直接更新する。

### 2-4. フロントエンド側の変更

- `FilterDrawer.tsx` の `AREA_GROUPS` に欠落している3市町村（桑折町・国見町・飯舘村）を追加する。所属グループは中通り（桑折町・国見町は県北、飯舘村は相双＝浜通り扱いに準拠する既存グループ定義に合わせる）。
- `AreaSelectMap.tsx`・`SpotFilterDrawer.tsx` は変更不要（既に59市町村ベース）。
- `RestaurantFilterDrawer.tsx` にはエリア絞り込みUIが存在しないため、今回は追加しない（スコープ外）。

### 2-5. バックエンドAPI側の変更

- `events_controller.rb` の `areas[]` パラメータ処理（IN句）はそのまま使えるため、ロジック変更は不要。許可される値の範囲が `Event::AREAS` のバリデーション拡張により自動的に広がる。
- `spots_controller.rb` / `restaurants_controller.rb` の検索パラメータ拡張（`municipality` ベースのサーバーサイド絞り込み追加など）は、Phase 4-B（横断検索UI）の設計時に必要に応じて検討する。今回は行わない。

## 3. スコープ外（今回は対応しない）

- Spot / Restaurant の `area`（7地方区分）列の削除・統合
- `RegionConquest`（制覇コレクション機能）が使う11地域グルーピング（kenpo / koriyama / sukagawa 等）の変更
- Restaurant のエリア絞り込みUIの新設
- Spot / Restaurant のサーバーサイド検索・ページネーション対応（Phase 4-Bで検討）

## 4. テスト方針

- **バックエンド**: Event / Spot / Restaurant 各モデルのバリデーションスペック（59市町村が許可される・範囲外文字列が拒否される）を更新・追加。`bundle exec rubocop` / `bundle exec rspec` を実行し 0 offenses・全テストパスを確認。
- **フロントエンド**: `npm run lint` で 0 errors / 0 warnings を確認。FilterDrawer に追加した3市町村が選択でき、選択後にイベント検索結果が正しく絞り込まれることをブラウザで確認（デフォルトテーマ・写真テーマ両方）。
- **手動データ移行の確認**: rake task 実行後、`Event.where(area: 'その他').count` が0件になることを確認。

1. `Event::AREAS`、`Spot`/`Restaurant` のバリデーションをコード変更

4. RuboCop / RSpec / ESLint 確認
5. ローカルで動作確認（イベント検索・スポット検索の両方でエリア絞り込みが正しく機能するか）
6. PR作成 → マージ → 本番デプロイ → 本番動作確認
