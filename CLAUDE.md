# Roami プロジェクト ガイドライン

## ユーザー情報

- エンジニア未経験（RaiseTech在籍中、ポートフォリオとして開発中）
- OS: Windows 11 / エディタ: Cursor（Claude Code拡張入り）

## 応答・説明スタイル（厳守）

- **すべての応答・コメント・説明は日本語で行うこと**
- 操作・コマンドを伝えるときは「裏で何が起きているか」を平易な言葉で必ず説明する
- 専門用語には必ず平易な補足をつける
- 「なぜそうするのか」の理由を省かない
- 手順は番号付きで一つずつ示す

## プロジェクト概要

**アプリ名:** Roami（ろーみー）  
**概要:** 福島での個人予定・おでかけを管理し、地域イベントも合わせて発見できる個人スケジュール管理Webアプリ

## 技術スタック

| 役割 | 技術 |
|------|------|
| フロントエンド | Next.js + TypeScript（App Router） |
| バックエンド | Ruby on Rails（APIモード） |
| データベース | MySQL（Docker） |
| 認証 | Devise（Rails gem） |
| 外部API | Connpass API（イベント自動取得） |
| カレンダー表示 | FullCalendar |
| デプロイ | AWS EC2 + RDS |

## サーバーポート（変更禁止）

| サーバー | ポート |
|----------|--------|
| バックエンド（Rails） | **8080** |
| フロントエンド（Next.js） | **3000** |
| データベース（MySQL） | **3306** |

別ポート起動・設定ファイル書き換えは禁止。起動手順は `/server-start` スキルを参照。

## GitHubワークフロー（厳守）

**開発フロー（順序厳守）:**  
`Issue作成 → 実装（ブランチ） → ローカル動作確認 → PR作成 → マージ → デプロイ → 本番確認`

- **作業前に必ずIssueを作成する**
- **ローカルで動作確認してからPRを作成する（確認前のPR作成禁止）**
- **本番デプロイはマージ後に行う（デプロイ→マージの逆順は禁止）**
- ブランチ命名: `{type}/{説明}-#{Issue番号}`（例: `feature/auth-#20`）
- コミットメッセージ: `{type}: {日本語説明}`
- **mainへの直接プッシュ禁止 / PRなしでmainにマージ禁止**
- PRマージはユーザーの確認後に行う

## 品質チェック

品質チェック作業はすべて **1つのIssue + 1つのPR** でまとめて対応すること。

### フロントエンド

```powershell
cd frontend
npm run lint
# → 0 errors, 0 warnings を確認
```

| 確認観点 | 内容 |
|---|---|
| `react-hooks/exhaustive-deps` | `useEffect` の依存配列に漏れがないか |
| `react/jsx-key` | `.map()` 内の要素に `key` があるか |
| 型安全性 | `any` 型を使っていないか |

### バックエンド

```powershell
cd backend
bundle exec rubocop
bundle exec rspec
# → 0 offenses / すべてのテストがパスすることを確認
```

| 確認観点 | 内容 |
|---|---|
| RuboCop | コーディング規約違反がないか |
| N+1 問題 | `includes` / `eager_load` を使っているか |
| 例外ハンドリング | `rescue_from` でエラーレスポンスを統一処理しているか |
| ストロングパラメータ | `permit` で明示しているか |

### ドキュメント整合性

> **ドキュメントは実装を「正」として修正する。実装をドキュメントに合わせない。**

`docs/requirements.md` / `docs/design.md` / `docs/database-design.md` を実装に合わせて更新する。

## 注意事項

- Webスクレイピングは使用しない（法的リスクのため）
- 外部データ取得はConnpass公式APIのみ使用する
